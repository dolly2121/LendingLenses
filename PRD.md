## 1. Executive summary

LendingLens is a working demonstration of a modern lakehouse pattern for a non-bank lender. It implements a Medallion architecture (Bronze, Silver, Gold) where one curated Gold layer serves two consumers simultaneously: a business intelligence dashboard and a Python AI service that analyses customer calls. It also demonstrates data quality gating and PII masking as first class pipeline steps.

The demo is a deliberate thin slice of the platform described in Liberty Financial's Data Engineer role: Microsoft Fabric feeding Power BI on one side, Python AI services on AKS on the other, with a shared trusted data layer in between. Everything runs locally. Every local component maps one to one to its production equivalent, and that mapping is documented.

## 3. Business scenario

A non-bank lender receives two data streams:

1. **Loan applications** that must be validated, cleaned, masked, and served to business dashboards.
2. **Customer call transcripts** that must be analysed for hardship and complaint signals, which risk and compliance teams care about deeply in Australian consumer lending.

Both outputs must come from the same Gold tables, so the business and the AI services share a single source of truth.

## 4. Functional requirements

| ID | Requirement |
|---|---|
| FR1 | Ingest raw loan records and call transcripts into Bronze, unmodified, with ingest metadata |
| FR2 | Validate loans in Silver against a declared schema. Failing records are quarantined with a reason, never silently dropped |
| FR3 | Mask direct identifiers in Silver. Customer names are hashed. Gold carries no direct identifiers |
| FR4 | Derive `risk_band` in Silver from documented business rules |
| FR5 | Log every quality check to a `dq_audit` table (check name, pass count, fail count, run id, timestamp) |
| FR6 | Publish two Gold tables: `gold_loan_summary` and `gold_call_insights` |
| FR7 | Dashboard renders loan metrics, call insights, and quality results, reading only Gold and audit tables, refreshing automatically |
| FR8 | AI service exposes `POST /process-call`: accepts a transcript, scores sentiment, sets hardship and complaint flags, appends to `gold_call_insights` |

## 5. Non-functional requirements

### 5.1 Demo NFRs (built and enforced)

| ID | Requirement |
|---|---|
| N1 | Entire system starts with one command from a fresh clone |
| N2 | Full pipeline run completes in under 60 seconds on a laptop |
| N3 | The live sequence (post a call while the dashboard refreshes) survives 5 consecutive rehearsals without error |
| N4 | Zero real data. All records are synthetic and generated by code in the repo |

### 5.2 Production NFRs (documented, not built)

These are addressed in `ARCHITECTURE_MAPPING.md` as the path to production, and exist to answer "how does this scale" questions credibly:

- **Scale**: millions of rows via Delta partitioning, incremental watermark loads, and Fabric Spark compute
- **Security**: Entra ID identity, workspace RBAC, encryption in transit and at rest, secrets in Key Vault
- **Privacy**: PII classification per column, masking at Silver, aggregate-only Gold, aligned with the Australian Privacy Principles
- **Governance**: data contracts, lineage, and discoverability via Microsoft Purview, semantic models for BI consumption

## 7. Out of scope

Real Fabric, AKS, or Power BI. Authentication. Streaming ingestion. Real customer data of any kind. Model training. Anything not listed in FR1 to FR8.
